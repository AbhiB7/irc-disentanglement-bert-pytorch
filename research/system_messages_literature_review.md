# How Papers Handle System Messages and Non-Dialogue Data on the Ubuntu IRC Dataset

**Context for this document:** This is a targeted literature summary for a BERT cross-encoder (Study 2) replicating and extending Kummerfeld et al. (ACL 2019) on the Ubuntu IRC conversation disentanglement dataset. The decision at hand is whether to filter, compress, or keep system messages (join/quit/nick-change events) during training and evaluation. This document contains only information directly relevant to that decision.

---

## What Are System Messages in This Dataset?

System messages are non-human utterances auto-generated by the IRC server. In the Ubuntu IRC `.ascii.txt` files, they appear as lines like:

```
02:26 system  ===zelot joined the channel
```

They have a fixed speaker field (`system` or similar) and stereotyped text patterns (`has joined`, `has quit`, `is now known as`, mode changes, topic changes). They contain no conversational content.

---

## The Gold Annotation Convention: System Messages Are Self-Links

**This is the most important fact.** The gold `.annotation.txt` files annotate system messages as **self-links** — that is, a message that replies to itself, with no human parent. This is not informal handling; it is a deliberate annotation decision made by Kummerfeld et al. (2019) and documented in their corpus guidelines.

The consequence is that system messages *do have gold labels in the annotation file*. They are not absent from the annotations. Removing them from training therefore removes gold-labelled examples, and removing them from evaluation means the evaluation script (`graph-eval.py`) will penalise you for missing predicted links — directly lowering reported F1.

## Paper-by-Paper Evidence

### 1. Kummerfeld et al. (ACL 2019) — The Baseline

**Preprocessing:** No filtering of system messages. All messages including system events are kept in training and evaluation.

**Handling:** System messages are annotated as self-links and the feed-forward model learns to predict them. Because their text is lexically distinctive (stereotyped patterns), the model learns this category easily.

**Relevant result:** The FF model achieves near-perfect performance on system message self-link prediction:

| Link Type         | Precision | Recall | F1       |
| ----------------- | --------- | ------ | -------- |
| System Messages   | 99.0      | 100.0  | **99.5** |
| Start of Topic    | 75.0      | 80.0   | 77.5     |
| Isolated Messages | 66.7      | 80.0   | 73.3     |

**Implication for your project:** The model does not need help with system messages — they are the easiest category. There is no computational or accuracy motivation to filter them from the perspective of what the model struggles with.

---

### 2. Yu & Joty (EMNLP 2020) — Pointer Networks

**Full citation:** Tao Yu and Shafiq Joty. "Online Conversation Disentanglement with Pointer Networks." EMNLP 2020.

**Preprocessing:** No filtering of system messages. Full dataset used.

**Handling:** Identical to Kummerfeld et al. — system messages are kept and annotated as self-links. Yu & Joty are the only paper to explicitly analyse self-link composition in detail (Table 3 of their paper):

| Self-Link Type        | Percentage of All Self-Links |
| --------------------- | ---------------------------- |
| System Messages       | 41%                          |
| Start of Conversation | 36%                          |
| Isolated Messages     | 33%                          |

**Critical finding:** System messages are NOT the only or even the majority source of self-links. 59% of self-links are regular human messages (conversation starters and isolated messages). This means that even from a "self-link detection" perspective, system messages are one category among several — not a special case that warrants separate treatment.

**Self-link handling strategy:** Yu & Joty introduce a **threshold-based self-link adjustment during decoding** — not a preprocessing filter. During inference, if the argmax prediction is a self-link but its probability is below a threshold, the model falls back to the next-highest candidate. This threshold is tuned on the dev set. This is an inference-time strategy, applied equally to all self-link types including system messages.

**Result of self-link threshold adjustment (cluster F1):**

| Model                 | Without Threshold | With Self-Link Threshold |
| --------------------- | ----------------- | ------------------------ |
| FF                    | 40.1              | 41.0                     |
| Ptr-Net               | 36.0              | 41.5                     |
| Ptr-Net + Joint Train | 39.7              | 44.5                     |

This shows the payoff from correctly handling self-links at inference time — but through threshold tuning, not through filtering or compression in preprocessing.

### 3. Pappadopulo et al. (DAG-LSTM, *SEM 2021)

**Full citation:** Duccio Pappadopulo, Lisa Bauer, Marco Farina, Ozan Irsoy, Mohit Bansal. "Disentangling Online Chats with DAG-Structured LSTMs." *SEM 2021.

**Preprocessing:** No filtering of system messages. Full dataset used including all message types.

**Handling:** System messages remain in the data as self-links. The DAG-LSTM also implements self-link threshold tuning at inference time (two thresholds: θ for argmax probability and δ for top-2 probability gap), mirroring the Yu & Joty approach.

**Self-link threshold results:**

| Model                              | Self-Link P | Self-Link R | Self-Link F1 |
| ---------------------------------- | ----------- | ----------- | ------------ |
| BiLSTM                             | 79.6        | 94.6        | 86.5         |
| DAG-LSTM                           | 82.8        | 93.8        | 88.0         |
| DAG-LSTM + threshold               | 87.7        | 92.4        | **90.0**     |
| DAG-LSTM + thread enc. + threshold | 89.8        | 90.6        | **90.2**     |

**Specific preprocessing note:** The DAG-LSTM paper does apply one form of context window management — they chunk each training conversation into contiguous 50-post windows, each with a 100-post past context and 50-post future context. For dev/test, they drop the first 900 posts (which are unannotated). This is *context window management for sequence modelling*, not message-type filtering. System messages within annotated windows are kept.

---

### 4. Zhu et al. (AAAI 2020) — Masked Hierarchical Transformer

**Full citation:** Henghui Zhu, Feng Nan, Zhiguo Wang, Ramesh Nallapati, Bing Xiang. "Who Did They Respond To? Conversation Structure Modeling Using Masked Hierarchical Transformer." AAAI 2020.

**Preprocessing:** No filtering of system messages. Uses full Kummerfeld et al. dataset. Reported graph-link F1: 70.6 (slightly below baseline FF 72.3).

**Handling:** Full dataset with self-link annotations intact. BERT-based utterance representations. This is the paper most architecturally similar to your Study 2 cross-encoder — they use contextual embeddings on the same pairwise task with the same evaluation metric.

**Implication:** Zhu et al. achieved 70.6 F1 with BERT alone (no handcrafted features), falling below the GloVe+FF baseline. This is a known result in your study2_status.md (B1 expected ~47 F1 for their specific cross-encoder variant). The dataset they used included system messages with no special handling.

---

### 5. Li et al. (DialBERT, 2020)

**Preprocessing:** No filtering. Full dataset including system messages. Cluster-level F1: 44.1.

## Consensus Across All Papers

| Paper                   | System Messages Filtered? | How Handled                                     |
| ----------------------- | ------------------------- | ----------------------------------------------- |
| Kummerfeld et al. 2019  | No                        | Self-link annotation, model learns them         |
| Yu & Joty 2020          | No                        | Self-link annotation + inference-time threshold |
| Pappadopulo et al. 2021 | No                        | Self-link annotation + inference-time threshold |
| Zhu et al. 2020         | No                        | Self-link annotation, no special handling       |
| Li et al. 2020          | No                        | Self-link annotation, no special handling       |

**No paper on this dataset has filtered, removed, or compressed system messages from training or evaluation.** The field-wide convention is to keep them as self-link labelled examples.

## Decision-Relevant Summary

**Option: Remove system messages entirely from training and evaluation**

- Not done by any paper on this dataset
- Removes gold self-link annotations → `graph-eval.py` penalises missing predictions
- Makes your results incomparable to all published benchmarks
- Verdict: **Do not do this**

**Option: Compress multiple system messages into one placeholder**

- Not done by any paper
- Corrupts positional and temporal features (`positiondist`, `timediffmin`)
- Creates distribution mismatch between training and inference
- Verdict: **Do not do this**

**Option: Skip system messages as parent candidates for regular messages only**

- Safe and architecturally sensible — no regular human message replies to a system event
- Applied at *pair generation* time, not at message-removal time
- The system message itself still generates a self-link pair (label=1) and gets a gold annotation
- Must be applied identically in `dataloader.py` (training) and `infer.py` (inference)
- Verdict: **Acceptable, but not done explicitly in any paper — the models learn this naturally**

**Option: Keep system messages, add self-link threshold tuning at inference**

- The approach used by Yu & Joty 2020 and Pappadopulo et al. 2021
- Addresses the actual problem: models over-predict self-links, hurting cluster F1
- Applied at inference time, not preprocessing time
- Fully consistent with all benchmarks
- Verdict: **Best practice per the literature**

**The correct framing of the supervisor's concern:** The issue is not that system messages are noise — they are the *easiest* category the model handles (99.5% F1). The actual challenge is that *all three self-link types* (system messages + conversation starters + isolated messages) collectively hurt cluster F1 when mis-predicted. The literature solution is inference-time threshold tuning on self-link probability, not preprocessing changes.
